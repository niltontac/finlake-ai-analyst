## Summary

A feature OBSERVABILITY adiciona dois componentes de instrumentação ao agente finlake-analyst que até então rodava completamente às cegas:

**LangFuse traces** — `CallbackHandler` por mensagem no `app.py`, com `session_id`, `question` e `retry_count` visíveis no LangFuse Cloud. O SQL gerado é capturado automaticamente nos spans de `generate_sql`. O `retry_count` é adicionado como score pós-loop via `Langfuse().create_score()`. Toda a lógica de tracing é protegida por `try/except` — falhas do LangFuse não interrompem o agente.

**DeepEval evals** — script standalone `evals/eval_sql.py` com 5 test cases (P1–P5) e `GEval` (LLM-as-judge usando Claude Sonnet 4.6). Roda com `uv run python evals/eval_sql.py`, imprime score por query e geral, e retorna exit code 0 se ≥ 3/5 queries passam.

## Lessons Learned

### Processo

- **Verificar versões instaladas no INÍCIO do DESIGN, não no Build.** O BRAINSTORM e o DEFINE assumiram langfuse 2.x e deepeval 1.x. A descoberta no DESIGN (via `uv pip show langfuse`) de que as versões instaladas eram 4.7.1 e 4.0.5 mudou toda a arquitetura. Se essa descoberta tivesse acontecido durante o Build, seria um blocker. Adicionando como regra: o primeiro passo de qualquer DESIGN que depende de pacotes externos deve ser `uv pip show <pacote>` para confirmar a versão real.

- **`inspect.getsource()` é mais rápido que documentação para APIs de terceiros.** A API do langfuse 4.x não estava completamente documentada no contexto de LangGraph. Inspecionar `CallbackHandler.__init__`, `on_chain_start` e pesquisar `last_trace_id` no código-fonte levou < 5 minutos e foi conclusivo. Prioritário para pacotes em evolução rápida.

- **O DESIGN gerou 5 ADRs fundamentados em evidência real de inspeção de API.** Isso foi possível porque o DESIGN dedicou tempo a investigar antes de escrever patterns. O Build foi linear como resultado.

### Técnico

- **LangFuse 4.x mudou para OpenTelemetry como backend.** O módulo de integração LangChain migrou de `langfuse.callback` para `langfuse.langchain`. Credenciais não são mais passadas no construtor do `CallbackHandler` — vêm de env vars (`LANGFUSE_PUBLIC_KEY`, `LANGFUSE_SECRET_KEY`, `LANGFUSE_HOST`). O atributo `last_trace_id` retorna o trace ID após execução. `Langfuse().create_score(trace_id=...)` funciona fora do contexto de trace para enriquecimento pós-loop.

- **`os.environ.setdefault()` é o padrão correto para bridging pydantic-settings → pacotes que leem env vars.** Pydantic-settings carrega o `.env` para o modelo mas NÃO popula `os.environ`. Usar `setdefault` (não `os.environ[key] = value`) garante que variáveis reais de produção não sejam sobrescritas.

- **DeepEval 4.x: `LLMTestCaseParams` → `SingleTurnParams`.** O import antigo funciona mas gera `DeprecationWarning`. `GEval.measure()` é síncrona; deepeval aplica `nest_asyncio` automaticamente ao ser importado, permitindo chamadas dentro de `asyncio.run()`.

- **`AnthropicModel` no deepeval evita dependência de `OPENAI_API_KEY`.** Disponível em `deepeval.models.llms.anthropic_model`. Usa o mesmo modelo configurado em `Settings` — sem chave extra.

- **Captura de `final_sql` e `final_retry_count` de eventos `on_chain_end` de nós LangGraph:** A estrutura esperada é `event["data"]["output"]["sql"]` e `event["data"]["output"]["retry_count"]`. Implementado com `.get(key, fallback)` — se o campo não existir, o valor anterior é preservado (graceful). Validação final requer smoke test com banco real (A-003/A-004 do AGENT_CORE ainda pendentes).

### Ferramentas e Bibliotecas

- **langfuse 4.7.1**: `langfuse.langchain.CallbackHandler` + `Langfuse().create_score()` + `flush()`
- **deepeval 4.0.5**: `GEval` com `SingleTurnParams` + `AnthropicModel` + `metric.measure()` síncrono
- **LangGraph 1.2.4**: `config={"callbacks": [...], "metadata": {...}}` para passar handlers via `astream_events`
